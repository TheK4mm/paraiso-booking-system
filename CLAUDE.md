# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Hotel Paraíso v2.1 — hotel platform. Spring Boot 4.1 (Framework 7, Security 7, Jackson 3) / Java 17 full-stack app: **public guest portal** (landing at `/`, booking wizard `/reservar`, guest lookup `/consulta-reserva`, client area `/mi-cuenta`) + REST API (`/api/...`, HTTP Basic, staff-only) + server-rendered Thymeleaf back-office (form login) over one service layer. PostgreSQL with Flyway migrations. Code identifiers, comments, commit messages, and the domain language are all in **Spanish** — follow that convention.

Boot 4 is modularized: integrations need their own module (e.g. `spring-boot-flyway`), slice-test annotations live in per-module `-test` artifacts (`org.springframework.boot.data.jpa.test.autoconfigure.DataJpaTest`, `...webmvc.test.autoconfigure.AutoConfigureMockMvc`, `...jdbc.test.autoconfigure.AutoConfigureTestDatabase`, `...jpa.test.autoconfigure.TestEntityManager`), Testcontainers is NOT managed by the Boot BOM (imported explicitly in dependencyManagement), and Jackson 3 dropped `write-dates-as-timestamps` (ISO-8601 is the default).

## Commands

```bash
mvn spring-boot:run                # run (http://localhost:8080, profile dev by default)
mvn clean package -DskipTests      # build jar
mvn test                           # unit tests (Mockito)
mvn verify                         # + Testcontainers ITs (skip automatically without Docker)
mvn test -Dtest=ClassName#method   # single test
```

Database: local PostgreSQL, database `hotel_paraiso` must exist; Flyway creates/updates the schema on startup (`ddl-auto=validate` — never let Hibernate manage schema). Dev seed data comes from `db/seed-dev/` (profile dev only — kept OUTSIDE `db/migration` because Flyway scans that classpath location recursively and prod would otherwise apply the seeds too). To reset dev data: drop/recreate the database and restart.

Dev credentials: `admin`/`admin123` (ADMIN), `recepcion`/`recepcion123` (RECEPCIONISTA), `cliente@hotelparaiso.com`/`cliente123` (CLIENTE — guests log in with their **email**; `username = email` for CLIENTE accounts). Config via `application.yml` + `application-dev.yml`/`application-prod.yml`; secrets through env vars (`DB_URL`, `DB_USERNAME`, `DB_PASSWORD`, `REMEMBER_ME_KEY`, `APP_BASE_URL`).

## Architecture

**Package-by-feature** under `com.hotel.paraiso`: each module (`cliente/`, `empleado/`, `habitacion/` [includes TipoHabitacion], `servicio/`, `reserva/`, `facturacion/` [Factura+Pago], `dashboard/`, `security/`) contains its entity, repository (+Specs), Request/Response DTOs, MapStruct mapper, service, REST controller and view controller. `portal/` is the public face: `PortalController` (landing), `portal/reserva/` (booking wizard + guest lookup; hand-built view records, no MapStruct) and `portal/cuenta/` (client area `/mi-cuenta`). Cross-cutting code lives in `common/{audit,crud,exception,mapper,validation,web}`.

### Key patterns

- **Catalog CRUD**: Cliente, Empleado, Servicio, TipoHabitacion, Habitacion extend `common/crud/AbstractCrudService` (hooks: `beforeCreate/beforeUpdate` for uniqueness, `applyRelations` for FK resolution, `toDetalle` for counts). **Reserva, Pago, Factura deliberately do NOT** — they have dedicated services with domain logic.
- **Mapping**: MapStruct with `unmappedTargetPolicy = ERROR` (`common/mapper/MapperCentralConfig`) and `disableBuilder` — every new entity/DTO field must be mapped or explicitly ignored or the build fails. Reserva/Pago/Factura mappers are response-only; their entities are built in services.
- **Search**: `JpaSpecificationExecutor` + per-module `*Specs` classes composed with `Specification.allOf`; sort sanitized via `common/web/SortWhitelist`. REST returns `PageResponse<T>`.
- **N+1 discipline**: `@EntityGraph` (to-one only) on paginated `findAll`; `ReservaService.findDetalle` hydrates collections with 3 constant fetch queries (`findByIdConHabitaciones/ConServicios/ConPagosYFactura` — same persistence context merges them). Collection counts use aggregate queries, only in detail views.
- **Business codes**: `RES-yyyy-nnnnnn`/`FAC-yyyy-nnnnnn` come from PostgreSQL sequences (`nextCodigoSeq()`/`nextNumeroSeq()`) — never `count()+1`.
- **Concurrency**: room availability and payment balance are validated under `PESSIMISTIC_WRITE` locks (`findAllByIdForUpdate`, `findByIdForUpdate`); `@Version` on Reserva/Pago/Factura/Habitacion.
- **State machine** (`ReservaService.validarTransicionEstado`): PENDIENTE→CONFIRMADA→CHECKIN→CHECKOUT; CANCELADA from PENDIENTE/CONFIRMADA; NO_SHOW from CHECKIN. CHECKIN sets rooms OCUPADA; leaving CHECKIN frees them. Invalid transitions → `BusinessException` (422).
- **Factura state is derived**: `FacturaService.recalcularEstadoPorReserva` runs after every payment change (PENDIENTE/PAGADA_PARCIALMENTE/PAGADA). Payment limit = factura total (with IVA) if one exists, else reserva price. Descuento may not exceed subtotal.
- **Errors**: services throw `common/exception` types; `ApiExceptionHandler` (@Order(1), RFC 7807 ProblemDetail) covers REST; `MvcExceptionHandler` (@Order(2)) turns business errors into flash+redirect for views. 404→`ResourceNotFoundException`, 400→`BadRequestException`, 422→`BusinessException`, 409→integrity/optimistic-lock.
- **Auditing**: entities extend `AuditableEntity` (creadoEn/actualizadoEn/creadoPor/actualizadoPor via Spring Data auditing reading the SecurityContext). Business events publish `ActividadEvent` → persisted to `activity_log` AFTER_COMMIT only.
- **Security — three zones**: `security/SecurityConfig` — public (`/`, `/reservar/**`, `/consulta-reserva`, auth routes, `/verificar-email/**`, static), client (`/mi-cuenta/**` → `hasRole("CLIENTE")`), back-office (**`anyRequest().hasAnyRole("ADMIN","RECEPCIONISTA")`** — default-closed: new internal routes need no config change; new public routes need an explicit permitAll). Form login + sessions + remember-me (CSRF on), HTTP Basic + 401 entry point + CSRF-exempt for `/api/**` (API is staff-only; CLIENTE gets 403). Post-login routing via `RolAuthenticationSuccessHandler`: in-flight booking in session → `/reservar/datos`; SavedRequest; else CLIENTE→`/mi-cuenta`, staff→`/dashboard`. Both auth handlers are AJAX-aware (`Peticiones.esAjax` = `X-Requested-With: XMLHttpRequest`, sent by the portal's auth modal): success → `200 {"redirect": url|null}` (null ⇒ JS reloads the current page), failure (`RolAuthenticationFailureHandler`) → `401 {"mensaje": …}`; classic navigation keeps the historic redirects (`/login?error`, role-based destinations). `POST /registro` follows the same split in `AuthViewController`: AJAX renders `fragments/auth :: registroPane` (422, with BindingResult) or `:: registroExito` (200); non-AJAX keeps the standalone page flow. Roles are the `Rol` enum (ADMIN, RECEPCIONISTA, CLIENTE); CLIENTE accounts are born ONLY via public registration — `UsuarioService` blocks creating/converting them from admin. `@PreAuthorize("hasRole('ADMIN')")` guards `AbstractCrudService.softDelete` and `FacturaService.anular` as defense in depth. Deletes are soft (`activo` flag / estado CANCELADA/ANULADA).
- **Guest portal rules**: public availability uses `HabitacionRepository.findHabitacionesReservables` — ignores the physical DISPONIBLE/OCUPADA state (a room occupied today is bookable next month); excludes only MANTENIMIENTO/BLOQUEADA/inactive + reservation overlaps, the same rule `ReservaService` re-checks under lock at confirm. `ReservaPublicaService.confirmar` resolves the Cliente (authenticated → linked ficha; guest → get-or-create by email, never overwriting an existing ficha; document-under-another-email rejected) and **delegates to `ReservaService.create`** — never duplicate its logic. Wizard state between steps lives in the session (`RolAuthenticationSuccessHandler.SESION_RESERVA_EN_CURSO`, survives the login detour); the session price is never authoritative. Guest lookup uses one generic not-found message and `/mi-cuenta` ownership failures are 404 (never confirm foreign codes exist). Clients cancel only PENDIENTE/CONFIRMADA.
- **Email verification**: `VerificacionEmailService` mirrors `PasswordResetService` (SHA-256-hashed single-use token, 24 h, link via `log.warn` — SMTP extension point; base URL from `app.base-url`). The Cliente ficha is created/linked ONLY after verification; the pending ficha data travels in `tokens_verificacion_email` columns. `usuarios.email_verificado` is separate from `activo` (admin sanction) — `HotelUserDetailsService` disables the account on either.

### Frontend

Thymeleaf, layout-dialect. `layout/app.html` = sidebar+topbar admin shell (nav highlighting uses `currentPath` from `GlobalViewAttributes`); `layout/auth.html` = auth/error card; `layout/public.html` = public shell (marketing navbar with `sec:authorize` actions + footer + same toast pattern), styled by `static/css/public.css` (public-only components `.hp-public-*`, `.hp-hero`, `.hp-room-card`, `.hp-wizard-steps`, `.hp-booking-summary`, `.hp-account-nav` — inherits `--hp-*` tokens from app.css, loaded first). Public pages: `portal/{landing,consulta}.html`, `portal/reservar/{fechas,disponibilidad,datos,exito}.html` (step indicator via `fragments/wizard.html`), `mi-cuenta/{index,reservas,detalle,perfil}.html` (side menu via `mi-cuenta/nav.html`). Guest photos live in `static/img/` (hero, `tipos/`, `galeria/`; free-license, vendored). Per-entity back-office templates (`{modulo}/{lista,form,detalle}.html`) built on typed fragments:
- `fragments/campos.html` — form fields bound with `th:field`/`th:errors` via preprocessing (`*{__${campo}__}`); fragments are `entrada`/`seleccion`/`areaTexto`/`numero`/`multiselect` (names deliberately NOT HTML tag names — see Gotchas). **Use `th:insert` on the grid wrapper div** (`th:replace` would destroy the `col-md-*` class).
- `fragments/auth.html` — portal auth: `modal` (Bootstrap modal `#hpAuthModal`, included at the START of `layout/public.html`'s body for anonymous visitors only — the CSRF token of its login form creates the session, which fails on long pages once the response buffer is committed) + `loginPane`/`registroPane`/`registroExito`. The SAME panes are inserted by the standalone `/login` and `/registro` pages (staff 302 target, no-JS fallback) — never duplicate the form markup. Modal field ids are prefixed `hpAuth*` to avoid colliding with page forms. `static/js/auth-modal.js` submits both forms via fetch + `X-Requested-With` and swaps server-rendered fragments/JSON inside the modal; triggers are `[data-hp-auth="login|registro"]` anchors that keep their `href` as fallback.
- `fragments/tabla.html` — sortable headers + pagination that preserve the query string (`queryBase`/`queryBaseSort` model attrs).
- `fragments/badges.html` (estado→color+Spanish label), `fragments/estados.html` (empty states).
- Confirmations use the generic modal wired by `data-hp-confirm` on forms (see `static/js/app.js`); flash `success`/`error` render as toasts from `layout/app.html`.
- All assets are local (`static/vendor/`, Inter font in `static/fonts/`) — do not add CDN references. Design tokens in `static/css/app.css` (`--hp-*`).

### Gotchas

- **Never name a `th:fragment` after an HTML tag** (`input`, `select`, `textarea`, …): fragment expressions (`~{f :: nombre}`) are *markup selectors* and also match elements by TAG name, so a raw `<input>` anywhere in the file gets inserted as a second match — this caused every form field to render twice (and the browser then submits both, which Spring joins into `"valor,"`). `fragments/campos.html` fragments are named `entrada`/`seleccion`/`areaTexto` for this reason; `tabla :: th` only matches itself (documented in the file).
- Don't index model Maps by enum objects in SpEL (`map[enumVal]` breaks); key label maps by `name()` strings.
- Enum `@Column` needs explicit `length` matching the DDL CHECK columns.
- Lombok `@Builder.Default` values don't apply outside builders; MapStruct mappers use `disableBuilder`, so defaults like `activo=true` are set via `@Mapping(constant=...)` in `toEntity`.
- Never modify an applied Flyway migration (checksum); add a new `V<n>__*.sql`. Dev-only seeds go in `db/seed-dev/` numbered `V1000+` (never inside `db/migration` — recursive scan). Because seeds sit at V1000+, every new real migration is out-of-order on a populated dev DB → dev sets `flyway.out-of-order: true` (prod stays strict; it never has V1000+). Beware: devtools restarts apply new migrations immediately — finish writing a migration before triggering a rebuild, or the checksum will mismatch.
- View controllers bind `@ModelAttribute("request")` DTOs; on `BindingResult` errors re-render the form with the same model attrs (title, editId, select data via `addFormData`).
- `Usuario.cliente` is LAZY: anything used outside the service transaction (controllers/views) must load it via `UsuarioRepository.findConClienteByUsernameIgnoreCase` (fetch-join) or you get `LazyInitializationException`.
- Date-range DTOs implement `common/validation/RangoFechas` so `@RangoFechasValido` applies (validator is generic, reports on `fechaSalida`).
- `MvcExceptionHandler.handleNotFound` carries `@ResponseStatus(NOT_FOUND)` — without it the 404 view would ship with HTTP 200.
